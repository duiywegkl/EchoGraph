# EchoGraph 注意力机制优化

## 概述

EchoGraph 的注意力机制是为了解决知识图谱中的"注意力分散问题"而设计的。当角色卡包含大量实体和关系时，LLM 的注意力会被分散到不重要的信息上，导致对话质量下降。

## 问题描述

### 典型问题场景
以"巴比伦酒店"角色卡为例：
- **主角色关联了18个实体**
- **包含大量细节概念**：经营理论、研发方向、管理优化、扩张策略等
- **信息冗余严重**：很多概念对当前对话可能并不重要
- **上下文被稀释**：LLM需要处理所有信息，导致注意力分散

### 影响
1. LLM 响应质量下降
2. 关键信息被忽略
3. 对话不够聚焦
4. 响应时间增加

## 解决方案

### 核心组件

#### 1. EntityImportanceEvaluator (实体重要性评估器)
- **集体重要性评估**：基于对话历史中的提及频率和上下文相关性
- **整体重要性评估**：基于实体类型、关系数量和关系质量
- **聚合评分**：结合集体和整体重要性，计算最终评分

#### 2. AttentionAwareContextBuilder (注意力感知上下文构建器)
- **实体过滤**：只保留重要性超过阈值的实体
- **分层构建**：核心角色 + 核心关联 + 相关信息
- **长度控制**：限制最大上下文实体数量

### 评分机制

#### 实体类型权重
```python
entity_type_weights = {
    'character': 1.0,      # 角色最重要
    'location': 0.8,       # 地点次之
    'item': 0.7,          # 物品
    'concept': 0.4,       # 概念类实体权重较低
    'event': 0.6          # 事件
}
```

#### 关系类型权重
```python
relationship_weights = {
    '拥有': 0.8, '控制': 0.9, '管理': 0.8, '创立': 0.7,
    '位于': 0.6, '使用': 0.5, '遵循': 0.3, '研发': 0.4,
    '实施': 0.4, '计划': 0.2, '支持': 0.3, # ... 更多
}
```

## 使用方法

### 1. 在UI中使用

1. 打开 EchoGraph 应用
2. 切换到"知识图谱"页面
3. 点击"注意力分析"按钮
4. 选择要分析的角色
5. 点击"开始分析"查看重要性报告
6. 点击"获取优化上下文"查看过滤后的上下文

### 2. 程序化使用

```python
# 初始化带注意力机制的GRAG记忆系统
attention_config = {
    'collective_weight': 0.6,      # 集体重要性权重
    'holistic_weight': 0.4,        # 整体重要性权重  
    'importance_threshold': 0.3,   # 重要性阈值
    'max_context_entities': 15     # 最大上下文实体数
}

memory = GRAGMemory(
    hot_memory_size=10,
    entities_json_path="path/to/entities.json",
    attention_config=attention_config
)

# 获取优化后的角色上下文
optimized_context = memory.get_optimized_character_context("角色名称")

# 获取重要性分析报告
report = memory.get_entity_importance_report("角色名称")
```

## 配置参数

### 核心参数
- **collective_weight** (0.6): 集体重要性权重
- **holistic_weight** (0.4): 整体重要性权重
- **importance_threshold** (0.3): 重要性阈值，低于此值的实体将被过滤
- **max_context_entities** (15): 最大上下文实体数量

### 调优建议
- **严格过滤**：提高 `importance_threshold` 到 0.5，减少 `max_context_entities` 到 8
- **宽松过滤**：降低 `importance_threshold` 到 0.2，增加 `max_context_entities` 到 20
- **注重对话历史**：提高 `collective_weight` 到 0.8
- **注重图结构**：提高 `holistic_weight` 到 0.6

## 效果评估

### 量化指标
1. **过滤比例**：被过滤掉的实体百分比
2. **上下文长度减少**：字符数和行数的减少比例
3. **重要实体保留率**：核心实体的保留情况

### 质量指标
1. **对话相关性**：LLM 响应与当前对话的相关程度
2. **信息准确性**：关键信息的准确传达
3. **响应聚焦度**：回答的针对性和集中度

## 实际案例

### 巴比伦酒店案例
- **原始实体数**：20个
- **过滤后实体数**：12个
- **过滤比例**：40%
- **上下文长度减少**：约35%

### 优化效果
- 保留了核心角色"巴比伦酒店"的完整信息
- 保留了重要关联实体如"莉拉"、"催眠符文阵法"
- 过滤了低重要性的概念实体如"KPI考核制度"、"精确时间考核系统"
- LLM 能更好地聚焦于当前对话相关的信息

## 技术实现

### 算法流程
1. **数据加载**：从 entities.json 加载实体和关系数据
2. **重要性计算**：为每个实体计算集体和整体重要性
3. **阈值过滤**：移除低于阈值的实体
4. **排序选择**：按重要性排序，选择前N个实体
5. **上下文构建**：分层构建优化的上下文字符串

### 缓存机制
- 5分钟上下文缓存，避免重复计算
- 配置更新时自动清空缓存
- 支持手动禁用缓存进行实时分析

## 模式兼容性

### ✅ 支持的模式
- **本地测试模式**: 完全支持，数据存储在 `data/local_mode/entities.json`
- **酒馆模式**: 完全支持，数据存储在 `data/tavern_characters/{角色名}/sessions/current/entities.json`

### 🔧 技术实现
注意力机制在所有模式下都使用相同的配置参数：
```python
attention_config = {
    'collective_weight': 0.6,      # 集体重要性权重
    'holistic_weight': 0.4,        # 整体重要性权重
    'importance_threshold': 0.3,   # 重要性阈值
    'max_context_entities': 15     # 最大上下文实体数
}
```

### 📂 数据路径管理
- **本地模式**: MainWindow 初始化时配置
- **酒馆模式**: API服务器根据会话ID动态分配路径
- **自动适配**: GRAGMemory 根据传入的 `entities_json_path` 自动适配不同模式
